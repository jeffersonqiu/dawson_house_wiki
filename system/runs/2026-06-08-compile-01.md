   whole-unit Painting Works).

2. **Canonical room name: "Common Bedroom" (not "Guest Bedroom").** Sources used four different
   names for the second bedroom across documents: "Study Room/Guest Bedroom", "Bedroom 2",
   "Guest Bedroom", and "Common Bedroom". The Extractor explicitly recommended the
   Harness/Compiler "settle on canonical room names — probably matching the signed contract's
   terminology" (call-out #2). The signed Renovation Agreement — the legally binding,
   highest-authority source — uses "Common Bedroom", so that was adopted as canonical.
   `Common Bedroom.md` documents all 4 aliases and the rationale in an explicit naming-decision
   box so the mapping is traceable for anyone searching by an older name.

## Process notes / learnings for future Compiler runs

- **Approval gate.** All 6 proposals were sitting at "awaiting human approval" — the Compiler
  cannot write room/vendor/task notes (or price/status changes) until that gate clears
  (`write-safety.md`). The human/owner reviewed and approved all 6 in this session (with the
  two structural overrides above), recorded inline in each proposal's file via a new
  "**Status:** ✅ APPROVED …" line + "**Human decision notes:**" block — this gives a durable,
  in-repo audit trail of what was approved, when, and with what modifications, rather than the
  approval living only in chat history.
- **Merge-only / no placeholders honoured.** Every note created corresponds to a real,
  evidenced item/vendor/task/decision from the source documents — none are speculative
  placeholders. Item notes live at `Rooms/{Room}/{Item}.md` (not a global `Items/` folder),
  matching `note-creation.md`.
- **Schema fidelity.** All frontmatter fields were matched exactly against
  `system/schemas/item.yaml`, `vendor.yaml`, and `task.yaml` (including verbatim `price` strings,
  zero-padded `TASK-####` IDs, and the `confidence: high|medium|low` enum).
- **Cross-linking.** Wikilinks (`[[Note Name]]`) were added wherever an item/task/vendor/decision
  references another compiled note (e.g. tasks ↔ items ↔ rooms ↔ vendors, decisions ↔ items),
  so Obsidian's graph view reflects the real relationships.
- **Open follow-ups surfaced as Tasks, not silently dropped.** Things like the dining-table
  balance/delivery uncertainty (TASK-0007, lowest confidence) and the kitchen layout cascade
  (TASK-0006) were captured as open tasks rather than guessed at — keeping low-confidence
  inferences visible and actionable rather than baked into item records as fact.
