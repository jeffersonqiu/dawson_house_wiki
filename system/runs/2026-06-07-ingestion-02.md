## Learnings / Issues encountered

*(Written for whoever picks this up cold — these are the practical "gotchas" discovered while
walking the entire 04 Renovation tree this session and the prior one.)*

1. **`parentId = '<id>'` + `pageSize: 100` reliably lists an ENTIRE folder's contents in one
   `search_files` call.** Every folder encountered in the 04 Renovation tree had ≤9 items, so
   pagination was never needed — but always pass a generous `pageSize` up front rather than
   relying on the default, since the default may be much smaller and you won't notice truncation
   unless you check the returned count against `get_file_metadata` / visual folder inspection.

2. **`read_file_content` is consistently more complete than `search_files`'s `contentSnippet`.**
   The snippet returned inline by `search_files` is frequently TRUNCATED (observed ~3,000–5,000
   character cutoffs on long PDFs/Docs), while `read_file_content` returns the FULL extracted
   text. Always re-pull with `read_file_content` before writing a raw snapshot — never rely on
   the snippet alone for anything beyond a quick relevance check. This tool also transparently
   handles OCR for scanned PDFs, Google Docs/Sheets/Slides exports, AND text-bearing images.

3. **`download_file_content` IGNORES `exportMimeType` for non-Google-native binary files.**
   Tested directly on `02 Reno Sheet.xlsm`: requesting `exportMimeType: "text/csv"` returned a
   574,144-character base64 blob that — when decoded — started with the ZIP magic number
   `PK\x03\x04` and contained `xl/worksheets/sheet1.xml` paths: i.e., the RAW UNCONVERTED XLSM
   binary, not a CSV conversion. The `exportMimeType` param appears to only function for
   Google-native files (Docs/Sheets/Slides → real format conversion). For arbitrary binary
   formats (.xlsm, .docx, .zip, etc.) you'll just get the original bytes back, base64-encoded —
   useless for text extraction via this path. **If `read_file_content` errors with "unsupported
   mime type" on a binary office file, there is no read-only path to its structured content** —
   record it as a metadata-only entry and move on; do not waste calls trying export conversions.

4. **Empty string from `read_file_content` / `contentSnippet` = reliable "photo, no text" signal.**
   Sampled multiple WhatsApp JPEGs (By Luvina batch + bed-frame photo) — all returned empty
   strings, consistent with pure photographs. Conversely, when an image DOES contain a drawing/
   document/screenshot with text, `read_file_content` performs OCR and returns real text (see
   next point). Treat empty-string results as a strong (not absolute — always sample at least 1-2)
   signal that an image folder is photo-only and can be recorded as a metadata batch rather than
   pulled file-by-file.

5. **CRITICAL: filename collisions exist in this Drive — never identify a file by name alone.**
   Two COMPLETELY DIFFERENT files are both named "WhatsApp Image 2026-02-07 at 3.17.30 PM.jpeg":
   one (`165urO1lTRA6mVmz75xQAxCUIFLmZ5Njs`, in "02 Bed Frame + Mattress") is a photo with EMPTY
   extractable content; the other (`18mDV5eIeDnHE3O3XfoR-4UcZUVW9WZGI`, in "2. ZX/Kove Design"
   under IDs Archive) is a floor-LAYOUT-PLAN drawing with full readable OCR text (room labels:
   Balcony, Living Dining, Bedrm 3, Store, Common/Master Bath, Kitchen; "ZX Design — HOUSE TO
   HOME"). If you see a familiar filename reappear in a different folder, ALWAYS verify the file
   ID and re-pull content — do not assume it's the same file or skip it as a "duplicate."

6. **OCR degrades predictably on handwritten/annotated/dense-print documents.** The handwritten
   "BUD Studio Quotation" draft PDF (dated 13/3/26, with scratch notes/revised figures scrawled
   over a printed quotation) produced text with garbled fragments alongside clean printed text —
   readable enough to extract both the print AND paraphrase the handwriting, but expect gaps.
   The "Gain City Denza Lucky Draw" coupon-stub sheet (dense small print, many repeated similar
   blocks) was the worst case — text was heavily fragmented; only paraphrase + serial-number
   list could be salvaged. When OCR is this degraded, write what you can confidently transcribe,
   clearly mark sections as "OCR-degraded"/"paraphrased", and note the source PDF should be
   manually reviewed if exact figures matter.

7. **Multiple draft revisions of the same contract exist with DIFFERENT totals — don't assume
   the newest-looking one is authoritative.** Found THREE Bud Studio Appendix A figures: an
   early handwritten draft (~$77,725, dated 13/3/26, with draft-only line items later removed),
   a "rev2" + reformatted-spreadsheet pair (both $75,700.00, ~27/3/26), and the FINAL SIGNED
   CONTRACT ($72,805.00 — includes a late $1,000 discount). Always treat the executed/signed
   document (in "02 Signed Documents") as authoritative, and use draft revisions only to trace
   how scope/pricing evolved — cross-reference notes were added to each raw file pointing at
   the signed contract.

8. **Large tool results get auto-saved to a temp `.txt` file** (JSON schema
   `{content, id, mimeType, title}`) when they exceed the inline token budget — inspect these
   with `python3 -c "import json,base64; ..."` or `jq`/Bash rather than trying to read the whole
   blob inline. This is how the XLSM base64 payload was decoded and identified as a raw ZIP.

9. **Folder traversal order matters for efficiency**: walking `0 Archive` first surfaced the
   nested "02. IDs Archive" vendor-comparison folders (Senso Studio, ZX/Kove, Reno Studio,
   Curated Co) which turned out to contain FOUR competing renovation quotations — a
   structurally important discovery for the eventual vendor-comparison compilation. Always fully
   enumerate `Archive`/`Old`/`Backup`-style folders rather than deprioritizing them; they often
   hold comparison-shopping artifacts that establish important context (e.g., why Bud Studio was
   ultimately chosen over three other quotes).
