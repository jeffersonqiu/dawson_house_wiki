## Learnings / Issues encountered

*(Written for Harness to fold into `system/agents/extractor.md`.)*

1. **Schema `source` field works best as a semicolon-separated multi-path string** when a
   record is corroborated by multiple raw files (e.g. an item confirmed by both the Reno
   Sheet tracker AND a Gain City order PDF). The schema doesn't specify a list type for
   `source`, so I used `"path/a.md; path/b.md"` — consistent and grep-able, but Harness may
   want to formalize this convention (or switch `source` to `list` like `related_items`).

2. **Cross-document price discrepancies are common and usually explainable, not errors.**
   Three Gain City bundle items (Fridge/Washer-Dryer/Microwave) show different prices in the
   Reno Sheet tracker vs. the actual Sales Order line items — almost certainly rebate/discount/
   GST allocation across a combined invoice, not data entry mistakes. Recommend extractors
   ALWAYS record both figures with the discrepancy flagged rather than silently picking one
   — picking wrong would plant a wrong "fact" that's hard to unwind later.

3. **Design-review comment PDFs are a goldmine of `task`-shaped material that's easy to
   under-extract.** The 21-page "First Draft - Comments" PDF reads like idle client feedback
   on first pass, but on close reading contains: (a) FIRM decisions already made ("we won't
   have a dishwasher!", "let's go with option 1"), (b) genuinely OPEN follow-ups needing
   resolution ("wardrobe wood vs white — not sure", "mirror placement — open to suggestions"),
   and (c) layout puzzles spanning multiple items (stove/sink/oven/hood swap discussion). I
   had to read every page closely to separate these three categories — a quick skim would
   have missed (b) and (c) entirely, which are exactly the "follow-up" material the `task`
   schema exists for. Recommend future extractor passes always fully read review/comment-type
   documents rather than treating them as low-value supplementary material.

4. **Item "status" inference needs a clear rule — I used "ordered" only when a matching
   purchase-order/invoice document exists, "shortlisted" otherwise** (regardless of whether
   the Reno Sheet tracker said "Confirmed" — I treated "Confirmed" + a corroborating order
   doc as "ordered", and "Confirmed" alone, without an order doc, more cautiously). This
   worked cleanly here because ~7 of 18 items had completed Gain-City/Other-Furniture order
   paperwork and the rest didn't — but Harness should formalize: does "Confirmed" in the
   source tracker map to schema `status: ordered` even without independent corroboration?
   I erred toward requiring corroboration (hence several "Confirmed"-tagged tracker rows
   ended up rated "shortlisted" + medium confidence in my proposals) — a human reviewer
   should confirm whether that's the right level of skepticism or overly conservative.

5. **Room-naming is INCONSISTENT across sources and will need a canonical-name decision
   before compilation.** The same physical room is called "Study Room/Guest Bedroom"
   (moodboard v1), "Bedroom 2" (moodboard v2), "Common Bedroom" (signed contract — the
   legally authoritative document), and grouped as "Guest Bedroom" (furniture tracker).
   Likewise "Living/Dining" vs. separate "Living Room"/"Dining Room". I picked names that
   matched the item-tracker's grouping labels (since that's the densest item-bearing source)
   but flagged the ambiguity in every affected proposal. **Recommend Harness/Compiler settle
   on canonical room names — probably matching the signed contract's terminology, since
   that's the document that will govern as-built cross-references — before compiling, to
   avoid creating duplicate room folders for the same physical space.**

6. **A genuine "whole-house" / cross-cutting item category exists with no obvious home** in
   the `Rooms/{Room}/{Item}.md` layout mandated by `note-creation.md` (Aircon, Day Curtain,
   Painting Works all apply "All rooms" / "Whole House" per the source tracker, but no
   "Whole House" room folder pattern is sanctioned). I parked Aircon under Kitchen as a
   pragmatic placement and flagged it loudly — Harness may want to add an explicit rule for
   cross-cutting items (a "General"/"Whole House" pseudo-room? a `scope: whole-house` item
   field? folding into the room note's prose rather than a dedicated item note?).

7. **OCR-degraded source documents (handwritten annotations, dense small print) genuinely
   limit extraction confidence and should propagate into the `confidence` field, not just a
   footnote.** E.g. the "Other Furniture" dining-table order form had THREE conflicting price
   readings ($1,360 / $4,360 / $1,450) from OCR — I trusted the cross-referenced clean figure
   ($1,450, corroborated by the Reno Sheet) but rated the overall record's delivery-tracking
   sub-facts (balance amount, delivery date) as separately low-confidence since those weren't
   independently corroborated anywhere.

8. **Two large source files turned out to be effectively out-of-scope for item/vendor/task
   extraction**: `ownership-valuation-2026-06-07.md` (1,465 lines — pure mortgage amortisation
   table, House-Purchase scoped per its own folder path) yielded zero item/vendor/task
   records; `reno-sheet-xlsm-archive` and the two photo-batch files (`by-luvina-...`,
   `whatsapp-image-...-bed-frame`) are metadata-only stubs with no extractable content. Don't
   skip reading them (you need to confirm they're empty), but budget near-zero extraction
   time for files the ingestion run already flagged as "metadata-only" or "out of scope".

9. **Avoiding the prior stall**: wrote each proposal file immediately after drafting it
   (6 separate `Write` calls across the session, interleaved with reading source material in
   batches) rather than holding all 6 proposals + the run log in working memory and emitting
   one giant blob at the end. This run log itself deliberately omits YAML dumps — they live
   in the proposal files, which are the single source of truth for the actual record content.
